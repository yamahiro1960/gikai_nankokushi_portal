## 2. 認証の仕組み（Google OAuth）

### 採用方式

**Google Identity Services（B方式）** を採用しています。  
Supabase の認証機能は使わず、Google のアクセストークンを直接取得する方式です。

### ログインの流れ

```
①「Googleでログイン」ボタンをクリック
        ↓
② Googleアカウント選択画面（初回のみ同意画面）
        ↓
③ Google からメールアドレスを取得
        ↓
④ Supabase の member_directory テーブルで照合
        ↓
⑤ 一致 → セッション保存（localStorage）→ index.html へ遷移
   不一致 → 「登録されていません」エラー
```

### セッション情報（localStorage に保存）

```javascript
{
    memberId:         // 議員ID（例: 202306）
    memberName:       // 氏名（例: 山本康博）
    memberTitle:      // 役職（例: 議員）
    accessRole:       // 権限（管理者 / 一般）
    googleEmail:      // ログインしたGmailアドレス
    googleAccessToken: // Googleカレンダー連携用トークン
    loginTime:        // ログイン時刻
}
```

### Client ID

```
995727635041-4btvd387rc69h0agjq4ld4ko9jlmpuki.apps.googleusercontent.com
```

> この値は `auth-config.js` に設定されています。変更する場合は Google Cloud Console で新しい Client ID を発行し、`auth-config.js` の `googleClientId` を更新してください。

## 3. テストユーザー方式の運用ルール

### 現在の状態

Google Cloud Console のアプリは **「テスト中（未審査）」** の状態で運用しています。  
- 利用者は最大 **100人** まで登録可能
- 現在の議員・職員は約50人規模のため、上限内で問題なし

### テストユーザーとは

Google OAuth で当ポータルにログインできるGmailアカウントを事前登録する仕組みです。  
**登録されていないGmailでは「アクセスがブロックされました」エラーが表示されます。**

### テストユーザーの登録手順

1. [Google Cloud Console](https://console.cloud.google.com) にアクセス  
   （ログインアカウント: yamamoto.yasuhiro.japan@gmail.com）
2. 左メニュー「Google Auth Platform」→「対象」を開く
3. 「テストユーザー」セクションの **「ユーザーを追加」** をクリック
4. 登録するGmailアドレスを入力して保存

### 新しい利用者を追加するときの手順

1. **Google Cloud Console でテストユーザーを追加**（上記手順）
2. **Supabase の member_directory に議員情報を登録**（次章参照）

この2つが揃って初めてログイン可能になります。

## 4. ユーザー登録・管理手順

### Supabase の member_directory テーブル

利用者情報はすべてこのテーブルで管理します。

| カラム名 | 内容 | 例 |
|---------|------|-----|
| member_id | 議員ID（6桁） | 202306 |
| name | 氏名 | 山本康博 |
| gmail | GmailアドレスS（ログイン照合に使用） | yamamoto.yasuhiro.japan@gmail.com |
| faction | 会派名 | 南国会 |
| is_current | 現職フラグ（true/false） | true |
| access_role | 権限 | 管理者 または 一般 |

### 議員・職員を新規登録する方法

**方法①: user-management.html から登録（推奨）**
1. ポータルにログイン（管理者アカウントで）
2. 左メニュー「ユーザー管理」を開く
3. 「新規登録」から氏名・Gmail・役職を入力して保存

**方法②: Supabase Table Editor から直接登録**
1. [Supabase ダッシュボード](https://supabase.com/dashboard) にアクセス
2. Table Editor → member_directory
3. 「Insert row」で必要なカラムを入力して保存

### access_role（権限）の設定

権限を変更する場合は Supabase の Table Editor か SQL Editor で変更します。

```sql
-- 管理者に変更する例
UPDATE member_directory
SET access_role = '管理者'
WHERE member_id = '202306';
```

> PowerShell から REST API で日本語を送信する場合は文字化けが発生する場合があります。  
> その際は Supabase ダッシュボードから直接編集してください。

## 6. 初回ログイン時の注意事項

### 「Googleがこのアプリを確認していません」警告について

初めてログインするGmailアカウントでは、以下の警告が表示される場合があります。

```
【画面表示】
このアプリはGoogleで確認されていません
このアプリはまだGoogleによる審査が完了していません。
（続行する場合は「詳細」をクリックしてください）
```

**これは正常な動作です。以下の手順で続行してください。**

1. 「詳細」をクリック
2. 「（サイト名）に移動する（安全ではないページ）」をクリック
3. 「続行」をクリック

この操作は **同じGmailアカウントでは初回のみ** 必要です。2回目以降は表示されません。

> **利用者への事前説明が重要です。**  
> 初回ログイン前に「最初だけこの画面が出ますが、安全ですので『詳細』→『続行』で進んでください」と伝えてください。

### 2回目以降のログイン

2回目以降は Google アカウントを選択するだけでログインが完了します（同意画面・警告は表示されません）。

## 7. 代替ログイン方法（緊急時）

Google アカウントが使えない場合のために、コンボボックス（名前選択）による代替ログインを用意しています。

### 代替ログインの使い方

1. login.html の「Googleでログイン」ボタンの下にある  
   **「Googleが使えない場合はこちら ▼」** をクリック
2. 折りたたみメニューが開く
3. ドロップダウンから自分の名前を選択
4. 「ログイン（仮）」ボタンをクリック

> **注意**: 代替ログインはGoogle認証をバイパスするため、セキュリティレベルが低くなります。  
> あくまで緊急時の手段として利用し、通常時はGoogleログインを使用してください。

## 8. Google Cloud Console の管理

### 現在の管理アカウント

- **オーナー**: yamamoto.yasuhiro.japan@gmail.com（山本康博 個人Gmail）
- **プロジェクト名**: 南国市議会DXポータル（または作成時の名称）

### 主な設定箇所

| 設定 | 場所 | 内容 |
|------|------|------|
| テストユーザー追加 | 「Google Auth Platform」→「対象」 | ログイン許可するGmailを登録 |
| OAuth Client ID | 「認証情報」→「OAuth 2.0 クライアントID」 | Client IDの確認・再発行 |
| 有効なAPI | 「APIとサービス」→「有効なAPI」 | Google Calendar API が有効であること |

### 事務局向けの簡潔な変更手順

#### 1. ログインできる利用者を追加・削除する

1. Google Cloud Console にログイン
2. 対象プロジェクトを選択
3. 「Google Auth Platform」→「対象」を開く
4. 「テストユーザー」で対象のGmailを追加または削除する
5. 保存する

この操作で、Googleログインを許可する利用者を管理できます。

#### 2. Googleログインの設定内容を確認する

1. 「APIとサービス」→「認証情報」を開く
2. 「OAuth 2.0 クライアント ID」を開く
3. Client ID、承認済みのJavaScript生成元、設定内容を確認する

Googleログインが急に使えなくなった場合は、まずこの画面を確認します。

#### 3. カレンダー連携が使えるか確認する

1. 「APIとサービス」→「有効なAPIとサービス」を開く
2. 一覧に「Google Calendar API」があるか確認する
3. 無効な場合は有効化する

#### 4. 管理アカウントを変更する

1. 「IAMと管理」→「IAM」を開く
2. 新しい管理者アカウントを追加する
3. 役割を「オーナー」にする
4. 引き継ぎ完了後、不要になった旧アカウントを削除する

### 管理者向けの詳細な変更手順

#### A. テストユーザーを追加する詳細手順

1. Google Cloud Console にログインする
2. 画面上部のプロジェクト選択で対象プロジェクトを選ぶ
3. 左上メニューから「Google Auth Platform」を開く
4. 「対象」をクリックする
5. 「テストユーザー」欄まで下に移動する
6. 「ユーザーを追加」をクリックする
7. 追加したいGmailアドレスを入力する
8. 「保存」をクリックする
9. 追加後、そのGmailが一覧に表示されていることを確認する

#### B. テストユーザーを削除する詳細手順

1. Google Cloud Console にログインする
2. 対象プロジェクトを選ぶ
3. 「Google Auth Platform」→「対象」を開く
4. 「テストユーザー」の一覧を表示する
5. 削除したいGmailアドレスの行を探す
6. 削除またはごみ箱アイコンを押す
7. 保存して、一覧から消えていることを確認する

#### C. OAuth Client ID を確認・変更する詳細手順

1. Google Cloud Console にログインする
2. 対象プロジェクトを選ぶ
3. 「APIとサービス」→「認証情報」を開く
4. 「OAuth 2.0 クライアント ID」の一覧から対象のクライアントを選ぶ
5. 次の内容を確認する
        - 名前
        - 承認済みの JavaScript 生成元
        - 必要に応じて承認済みのリダイレクト URI
6. 必要な変更を行い、「保存」をクリックする
7. Client ID を再発行または差し替えた場合は、[auth-config.js](auth-config.js) の `googleClientId` も必ず更新する

#### D. Google Calendar API を有効化する詳細手順

1. Google Cloud Console にログインする
2. 対象プロジェクトを選ぶ
3. 「APIとサービス」→「ライブラリ」を開く
4. 検索欄で「Google Calendar API」を検索する
5. 対象APIを開く
6. 「有効にする」をクリックする
7. すでに有効な場合はそのままでよい

#### E. オーナーアカウントを引き継ぐ詳細手順

1. Google Cloud Console にログインする
2. 対象プロジェクトを選ぶ
3. 「IAMと管理」→「IAM」を開く
4. 「アクセスを許可」または「プリンシパルを追加」をクリックする
5. 新しい管理者用メールアドレスを入力する
6. 役割を「オーナー」に設定する
7. 保存する
8. 新しい管理者アカウントでログインできることを確認する
9. 問題がなければ旧アカウントの権限を「削除」または「権限を下げる」で整理する

#### F. 変更後に必ず行う確認

1. テストユーザーを追加した場合
        追加した本人が Google ログインできるか確認する
2. Client ID を変更した場合
        [auth-config.js](auth-config.js) の設定更新後、実際にログイン画面で動作確認する
3. オーナーを変更した場合
        新しい管理者アカウントだけで設定変更が可能か確認する

### 将来の対応（推奨）

現在はオーナーが個人Gmailですが、本番運用に向けて以下を推奨します。

1. 議会事務局専用の Google Workspace アカウントを作成
2. Google Cloud Console の IAM で専用アカウントをオーナーに昇格
3. 個人Gmailのアクセス権を削除

> IAM の変更は「Google Auth Platform」→「設定」→「IAMと管理」から行えます。

## 10. トラブルシューティング

### 「アクセスがブロックされました」が表示される

**原因**: Google Cloud Console のテストユーザーに登録されていない  
**対処**: Google Cloud Console → 「対象」 → 「テストユーザーを追加」で当該Gmailを登録

### 「このアカウントは登録されていません」が表示される

**原因**: member_directory にそのGmailアドレスが登録されていない  
**対処**: user-management.html または Supabase Table Editor で議員情報を登録

### Googleログインボタンが表示されない

**原因**: Google Identity Services（GIS）スクリプトの読み込み失敗（ネットワーク問題など）  
**対処**: ページをリロード。それでも解決しない場合は代替ログインを使用

### ログイン後に挨拶が表示されない・氏名がおかしい

**原因**: member_directory の `name` と `gmail` の対応が間違っている  
**対処**: Supabase Table Editor で当該レコードの `gmail` カラムを確認・修正

### 管理者なのにシステム設定に入れない

**原因**: `access_role` が `管理者` になっていない  
**対処**:
```sql
UPDATE member_directory
SET access_role = '管理者'
WHERE gmail = '該当のGmailアドレス';
```
Supabase SQL Editor で実行してください。
